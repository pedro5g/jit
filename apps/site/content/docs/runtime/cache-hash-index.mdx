---
title: Cache, hash and indexes
description: Understand identity caches, structural hashes and indexed repeated queries.
---

Schema-only operations use a `WeakMap` from schema identity to compiled function. Query
and mapper templates cache pure source while applying user bindings to a fresh closure.
This prevents one caller's values from leaking into another compiled operation.

`JIT.hash(schema)` emits a shape-specialized structural hash. Equal values under the same
schema produce the same hash; use it for memoization buckets and change detection, then
confirm collisions with compiled equality when correctness requires it.

Entity hints and keyed indexes are useful when the same list receives many
lookups. Index caches are weakly attached to source collections. A one-off scan
should stay a scan: building an index has an upfront time and memory cost.

## Cache tiers

Schema-only operations use `WeakMap<schema, function>`. Query/mapper plans cache
the pure source factory and apply each caller's bindings to a fresh closure.
This keeps compiled templates reusable without sharing user state.

## Hash strategy

Hashes are accelerators, not identity proofs. Use the hash to select a bucket,
then compiled equality to resolve collisions where correctness matters. Weak
hash caches are useful for immutable/reused objects; mutation invalidates the
assumption.

## Index break-even

An index costs O(n) build time and memory. It pays off only across enough
lookups or range scans. Measure cardinality and mutation rate: a frequently
rebuilt index over a short list can be worse than direct specialized scans.

`JIT.watchedList(..., { key })` maintains identity indexes while an aggregate
is edited, but it does not expose a version counter. Invalidate an external
cache when `isChanged()` becomes true or from the change arrays returned by
`snapshot()`. The [watched lists guide](/docs/runtime/watched-lists) explains
the state and identity contract.
